# 보안 정책

Stream은 아직 초기 단계의 라이브러리입니다. 프로젝트는 테스트, 검증 규칙, 패키징 검사, 워크플로우 검사, 문서 검토를 통해 이미 어느 정도 단단해졌지만, 보안 피드백, 정확성 피드백, 하드닝 제안은 초기에 자주 받을수록 좋습니다.

## 취약점 보고 방법

Stream에서 보안 문제가 있다고 판단되는 경우, 상세 exploit 정보, 비밀값, 토큰, 비공개 인프라 정보를 공개 공간에 그대로 게시하지 말아 주세요.

이 저장소의 대부분의 보고는 GitHub 기반 협업 방식을 권장합니다.

- 공개적으로 논의해도 안전한 약점, 위험한 동작, 누락된 validation boundary, 패키징 우려, 의존성 위험, 워크플로우 정책 문제, replay 또는 integrity 동작 문제는 GitHub issue로 열어 주세요.
- 이미 구체적인 하드닝 수정, 회귀 테스트, 워크플로우 개선, 문서 업데이트가 준비되어 있다면 pull request로 보내 주세요.
- 아직 공개되면 안 되는 민감한 내용이라면, 저장소의 보안 보고 채널이 활성화되어 있을 경우 그 경로를 사용해 주세요.

Issue, PR, 또는 비공개 보고에는 가능하면 다음 내용을 포함해 주세요.

- 문제에 대한 짧은 설명,
- 영향받는 버전 또는 커밋 범위,
- 재현 단계 또는 proof of concept,
- 예상되는 영향 범위,
- 문제가 append, scan, replay, export, integrity, rebuild, repair 중 어느 경로에 영향을 주는지.

이 문제가 보안 버그인지 정확성 버그인지 확신이 없더라도, 일단 issue를 열고 보안 관련 우려라고 명확히 표시해 주시면 됩니다.

## 지원 버전

보안 수정은 우선 가장 최신 개발 라인을 대상으로 진행합니다. 별도로 명시하지 않는 한, 이전 버전에 대한 백포트 수정은 제공되지 않을 수 있습니다.

## 특히 유용한 보고 유형

Stream에서는 다음과 같은 보고가 특히 유용합니다.

- 손상된 canonical history가 조용히 잘못 해석되는 경우,
- `strict` replay가 fail closed가 아니라 fail open처럼 동작하는 경우,
- `tolerant` replay가 gap이나 warning을 숨기는 경우,
- helper index가 source of truth처럼 취급되는 경우,
- repair 동작이 canonical history를 과하게 다시 쓰는 경우,
- example, CLI, 패키징, 문서에서 비밀값이나 위험한 기본값이 노출되는 경우,
- 로컬 trust boundary를 약하게 만드는 의존성, 워크플로우, 빌드 절차 문제.

## 공개 및 대응 절차

보고가 접수되면, 유지관리자는 가능한 한 다음 순서로 대응하는 것을 목표로 합니다.

1. 접수 확인,
2. 보고 내용 검증,
3. 수정 또는 완화 조치 준비,
4. 필요할 경우 회귀 방지 테스트 추가,
5. 필요할 경우 릴리스 노트나 관련 문서에 변경 사항 기록.
