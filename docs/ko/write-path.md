# 쓰기 경로

이 페이지에서는 레코드가 `Stream`을 입력하는 방법, 라이브러리에서 표준 기록으로 허용하는 내용, 추가 실행이 실제로 진행되는 방법, 안전하지 않은 가정 없이 추가 결과를 해석하는 방법에 대해 설명합니다. `Getting Started`에서 첫 번째 성공적인 쓰기가 표시된 경우 이 페이지에서는 해당 성공을 의미 있게 만드는 쓰기 측 규칙을 설명합니다.

여기서의 목표는 "`append()`에 전화해"라고 말하는 것뿐만이 아닙니다. 목표는 다음 사항을 결정하는 데 도움을 주는 것입니다.

- 도대체 무엇을 써야 하는가
- 입력의 모양은 무엇입니까?
- 추가 성공이 실제로 의미하는 것
- 배치 동작이 트랜잭션의 전부 아니면 전무 기대와 어떻게 다른지
- 내구성 모드가 반환된 결과의 의미에 미치는 영향
- 새 세그먼트가 생성되는 시기와 이것이 정상인 이유

## 이 페이지가 결정하는 데 도움이 되는 것

이 페이지를 읽고 나면 다음 질문에 명확하게 답할 수 있습니다.

- 허용 가능한 정식 기록으로 간주되는 것
- `append()` 또는 `append_many()`에 전화해야 하는지 여부
- 정규화와 검증이 상호 작용하는 방식
- `accepted`, `rejected`, `durability_status`, `durable_count`의 실제 의미
- 배치의 일부만 유효하면 어떻게 되나요?
- `flush`과 `fsync`의 운영상의 차이점
- 시퀀스, 세그먼트, 오프셋과 같은 배치 메타데이터를 추가하는 것이 중요한 이유

## 쓰기 경로에 자체 페이지가 필요한 이유

언뜻 보면 `Stream` 글쓰기가 사소해 보일 수 있습니다. 사전을 전달하고 결과를 다시 가져오는 것입니다. 실제로 쓰기 경로는 `Stream`이 데이터가 정식 기록이 될 수 있는지 여부를 결정하는 곳입니다.

즉, 쓰기 경로에는 몇 가지 중요한 경계가 적용됩니다.

- 임의 입력과 허용된 표준 레코드 사이의 경계
- 허용된 쓰기와 거부된 쓰기 사이의 경계
- 허용되는 쓰기와 지속적으로 커밋되는 쓰기 사이의 경계
- 표준 세그먼트 기록과 파생 도우미 상태 사이의 경계

대부분의 다운스트림 혼란은 쓰기 측 가정에서 시작되기 때문에 이는 중요합니다. 호출자가 "추가 성공"이 실제 의미보다 더 많은 것을 의미한다고 생각하거나 "추가 거부"가 일괄 처리에서 복구 가능한 항목이 없음을 의미한다고 믿는 경우 나중에 읽기 및 복구 동작이 놀랍게 느껴질 것입니다.

## 한 문장 쓰기 모델

`Stream`은(는) 입력을 정규화하고, 표준 형태에 대해 유효성을 검사하고, 추가 순서 배치를 할당하고, JSONL 세그먼트 항목을 작성하고, 매니페스트 상태를 업데이트하고, 파생 인덱스 포인터를 새로 고쳐 표준 레코드를 작성합니다.

그 문장은 미니어처의 쓰기 경로입니다.

## 쓰기 경로가 소유하는 것

`Stream`의 쓰기 경로에는 몇 가지 구체적인 책임이 있습니다.

- 지원되는 레코드 입력을 표준 매핑으로 정규화
- 표준 필드 및 페이로드 구조 검증
- 대상 세그먼트와 다음 시퀀스 번호를 선택하십시오.
- 표준 세그먼트 저장소에 단일 JSONL 항목 추가
- 추가 내구성 상태 기록
- 사전 매니페스트 상태
- 파생 인덱스에 포인터 항목 추가

중요한 점은 글쓰기가 단순히 "이 dict를 직렬화"하는 것이 아니라는 것입니다. "추가 순서 규칙에 따라 이 값을 표준 로컬 기록에 허용합니다."

## `Stream`에서 허용하는 입력

`Stream`은 유효한 표준 레코드로 정규화할 수 있는 입력을 허용합니다.

실제로 여기에는 다음이 포함됩니다.

- 지원되는 `Spine` 레코드 객체
- 이미 표준 필드를 전달하는 매핑과 유사한 Python 레코드
- 허용 가능한 매핑으로 정규화되는 데이터 클래스 입력

라이브러리는 직렬화할 수 있다는 이유만으로 임의의 Python 객체나 임의의 JSON 모양 blob을 허용하지 않습니다.

### 유효한 매핑 예

```python
record = {
    "record_ref": "record/eval-start",
    "record_type": "structured_event",
    "recorded_at": "2026-04-03T00:10:00Z",
    "observed_at": "2026-04-03T00:10:00Z",
    "producer_ref": "scribe.python.local",
    "run_ref": "run/eval-1",
    "stage_execution_ref": "stage/evaluate",
    "operation_context_ref": "op/evaluate-open",
    "correlation_refs": {"trace_id": "trace/eval-1"},
    "completeness_marker": "complete",
    "degradation_marker": "none",
    "schema_version": "1.0.0",
    "payload": {
        "event_key": "evaluation.started",
        "level": "info",
        "message": "Evaluation started.",
    },
}
```

### 잘못된 매핑 예

```python
invalid_record = {
    "record_type": "structured_event",
    "payload": {},
}
```

그 두 번째 기록은 단지 미학적 의미에서 불완전한 것이 아닙니다. 필수 필드와 페이로드 보장이 누락되어 정식 기록이 될 수 없습니다.

## 정규화가 검증보다 먼저 이루어집니다.

쓰기 경로에서 미묘하지만 중요한 부분 중 하나는 `Stream`이 지원되는 입력을 검증하기 전에 정규화한다는 것입니다.

실제로 이는 다음을 의미합니다.

- `Spine` 객체는 정식 페이로드 매핑으로 변환됩니다.
- 데이터 클래스 입력은 먼저 매핑으로 변환됩니다.
- 매핑 입력은 문자열 키 표준 구조로 정규화됩니다.

그 후에야 `Stream`은 정규화된 기록이 허용 가능한 표준 기록인지 여부를 확인합니다.

쓰기 경로는 의도적으로 입력 *형식*에 대해서는 관대하지만 정식 *의미*에 대해서는 엄격하기 때문에 이는 중요합니다.

다시 말해서:

- 여러 입력 형태가 허용될 수 있습니다.
- 표준 레코드 의미는 하나만 허용됩니다.

## 검증이 실제로 시행하는 것

`Stream`의 유효성 검사는 단순한 유형 확인이 아닙니다. 나중에 재생 가능한 기록을 해석할 수 있도록 모양과 안전 가정을 시행하고 있습니다.

여기에는 다음과 같은 검사가 포함됩니다.

- 필요한 안정적인 참조는 공백이 아니고 올바른 형식이어야 합니다.
- 타임스탬프는 이미 정규화되어 있어야 합니다.
- `recorded_at` 및 `observed_at`은 순서대로 이해되어야 합니다.
- 스키마 버전이 지원되어야 합니다.
- `record_type`은(는) 허용되는 정식 계열 중 하나여야 합니다.
- 페이로드 구조는 선택한 `record_type`과 일치해야 합니다.

잘못된 쓰기가 단순히 불편한 것이 아니기 때문에 이는 중요합니다. 만일 그것이 받아들여진다면 그것은 정식 역사를 오염시킬 것입니다.

## 레코드 유형은 쓰기 경로에 중요합니다.

쓰기 경로는 모든 레코드 페이로드를 교환 가능한 것으로 처리하지 않습니다.

예를 들어:

- `structured_event`은(는) `event_key`, `level` 및 `message`과 같은 이벤트 지향 필드를 예상합니다.
- `metric`에는 `metric_key`, `value` 및 집계 의미와 같은 측정항목별 필드가 필요합니다.
- `trace_span`은 범위 지향 타이밍 및 ID 필드를 기대합니다.
- 성능 저하 기록은 자체 페이로드 기대치를 따릅니다.

중요한 점은 `record_type`이(가) 설명적인 장식이 아니라는 것입니다. 페이로드 유효성 검사가 작동하는 방식이 변경되므로 저장소에서 허용하는 항목이 변경됩니다.

## 쓰기 파이프라인 단계별

높은 수준에서 성공적인 추가는 다음 단계를 통해 진행됩니다.

1. 입력을 정규화하다
2. 표준 매핑 검증
3. 추가 슬롯을 선택하세요
4. 세그먼트 항목 직렬화
5. 대상 세그먼트에 기록
6. 내구성 모드에 따라 플러시 또는 fsync
7. 매니페스트 진행 커밋
8. 파생 색인 항목 추가
9. 추가 배치 및 내구성 메타데이터 반환

이 순서는 추가 결과가 그토록 유익한 이유를 설명합니다. 이는 임의의 메타데이터가 아닙니다. 쓰기가 방금 저장소를 통과한 경로를 설명합니다.

## 추가 슬롯 선택

쓰기가 파일에 도달하기 전에 `Stream`은 추가 슬롯을 선택합니다. 해당 슬롯은 다음을 결정합니다.

- 타겟 `segment_id`
- 다음 `sequence`
- 해당 세그먼트 내의 다음 줄 `offset`

언뜻 보면 이는 내부 장부 관리처럼 들릴 수 있습니다. 실제로 이는 기록의 정식 배치의 시작입니다.

이후 작업은 배치 사실에 따라 달라지기 때문에 이는 중요합니다.

- 스캔은 추가 순서를 유지합니다
- 재생은 정렬된 기록을 해석합니다.
- 인덱스는 `(segment_id, offset)` 쌍을 다시 가리킵니다.
- 무결성은 시퀀스 연속성에 대한 이유를 확인합니다.

## 세그먼트 롤오버가 존재하는 이유

`Stream`은 모든 레코드를 무한히 증가하는 하나의 파일에 기록하지 않습니다. 현재 파일이 구성된 크기 임계값에 도달하면 세그먼트가 롤오버됩니다.

이는 건강한 추가 경로가 결국 다음과 같은 결과를 가져올 수 있음을 의미합니다.

- 공간이 있는 동안 현재 세그먼트를 계속 사용합니다.
- 구성된 크기 제한에 도달하면 새 세그먼트로 전환
- 세그먼트 경계를 넘어 시퀀스 번호가 증가하도록 유지

중요한 점은 새로운 세그먼트가 손상이나 조각화의 징후가 아니라는 것입니다. 이는 쓰기 경로의 일반적인 부분입니다.

## 세그먼트에 기록되는 내용

세그먼트 파일은 원시 호출자 입력을 저장하지 않습니다. 표준 세그먼트 항목을 저장합니다.

```json
{
  "sequence": 12,
  "appended_at": "2026-04-03T00:10:00Z",
  "record": {
    "...": "canonical record payload"
  }
}
```

이 구조는 다음을 구분하기 때문에 중요합니다.

- 스토어가 소유한 메타데이터 추가
- 호출자가 제공한 표준 레코드 데이터

추가 메타데이터는 현지 수용 여부를 알려줍니다. 표준 레코드는 관찰된 이벤트, 메트릭, 범위 또는 성능 저하 사실에 대한 내용을 알려줍니다.

## 매니페스트에 기록되는 내용

세그먼트 쓰기가 성공한 후 `Stream`은 매니페스트 상태를 발전시킵니다.

실제로 이는 다음을 업데이트하는 것을 의미합니다.

- `current_segment_id`
- `next_sequence`
- `last_committed_sequence`
- `current_segment_record_count`

매니페스트는 나중에 저장소가 현재 쓰기 경계가 있다고 믿는 위치를 추가하기 때문에 중요합니다. 해당 값이 관찰된 세그먼트 기록에서 벗어나면 무결성 검사를 통해 해당 불일치가 명시적으로 드러날 수 있습니다.

## 인덱스에 기록되는 내용

표준 기록이 작성되고 매니페스트 진행이 진행된 후에만 `Stream`에서 도우미 인덱스 항목을 추가합니다.

이러한 파생 항목은 다음과 같은 포인터 정보를 저장합니다.

- `sequence`
- `segment_id`
- `offset`

다음과 같은 실용적인 스캔 제품군에 맞춰져 있습니다.

- `run_ref`
- `stage_execution_ref`
- `record_type`

중요한 점은 인덱스가 정식으로 작성된 것이 아니라 쓰기 측 편의 아티팩트라는 것입니다.

## `append()` 대 `append_many()`

다음과 같은 경우 `append()`을(를) 사용하세요.

- 당신은 하나의 논리적 레코드를 쓰고 있습니다
- 호출 사이트 단순성은 일괄 처리보다 더 중요합니다.
- 목록을 직접 작성하지 않고 단일 레코드 배치의 반환 형태를 원합니다.

다음과 같은 경우 `append_many()`을(를) 사용하세요.

- 이미 여러 후보자 기록이 있습니다
- 부분 승인이 허용됩니다.
- 배치 결과를 설명하는 하나의 추가 결과를 원합니다.

작동상 `append()`은 동일한 쓰기 경로의 단일 레코드 형식일 뿐입니다. 다른 스토리지 모델을 생성하지 않습니다.

## 배치 동작은 트랜잭션이 아닙니다.

이는 쓰기 경로에서 가장 중요한 실제 사항 중 하나입니다.

`append_many()`은 전부 아니면 전무 거래가 아닙니다. 입력을 통해 반복되며 다음을 수행합니다.

- 검증된 기록을 수락합니다.
- 검증에 실패한 레코드를 거부합니다.
- 표준 역사에서 허용된 기록을 보존합니다.
- 발신자에게 거절을 다시 보고합니다.

이는 배치가 부분적으로 성공할 수 있음을 의미합니다.

개념적 결과는 다음과 같습니다.

```text
success == False
accepted_count == 2
rejected_count == 1
rejected[0] == "record[1] rejected: ..."
```

이는 호출자가 거절이 "아무것도 기록되지 않았다"는 의미라고 가정해서는 안 되기 때문에 중요합니다. `Stream`에서는 부분적인 성공이 정상적이고 명시적입니다.

## `AppendResult` 읽는 방법

결과 개체를 읽는 방법을 알고 나면 쓰기 경로를 추론하기가 훨씬 쉽습니다.

### `accepted`

`accepted` 튜플에는 성공적으로 추가된 레코드당 하나의 `AppendReceipt`이 포함됩니다.

각 영수증에는 다음 사항이 기재되어 있습니다.

- 할당된 `sequence`
- `segment_id`
- `offset` 라인
- `record_ref`
- `record_type`
- `run_ref`
- `durability_status`

### `rejected`

`rejected` 튜플에는 검증에 실패한 레코드에 대해 사람이 읽을 수 있는 거부 메시지가 포함되어 있습니다.

이것은 단순한 로그가 아닙니다. 이는 정식 기록에 인정되지 않은 내용에 대한 명시적인 쓰기측 기록입니다.

### `durability_status`

배치 수준에서 `durability_status`은 허용된 쓰기의 내구성이 얼마나 되었는지 요약합니다.

### `durable_count`

`durable_count`은 허용된 쓰기 수가 저장소에 구성된 내구성 임계값에 도달했는지 보고합니다.

### `success`

`success`은 배치에 거부가 없음을 의미합니다. 그 이상을 의미하지는 **않습니다**.

특히, 그 자체로는 다음을 의미하지 않습니다.

- 매장은 전반적으로 건강해요
- 인덱스는 신뢰할 수 있습니다
- 역사상 다른 ​​곳에서는 이전의 부패가 존재하지 않습니다.

## `DurabilityStatus`의 의미

쓰기 경로는 수용성과 내구성을 구분합니다.

가능한 상태는 다음과 같습니다.

- `accepted`
- `flushed`
- `fsynced`

실제 의미는 매장 구성에 따라 다릅니다.

### `flush`

저장소가 `DurabilityMode.FLUSH`을 사용하는 경우 추가 경로는 다음과 같습니다.

- JSONL 항목을 작성합니다.
- 파일 핸들을 플러시합니다.
- 허용된 쓰기에 대해 `flushed` 내구성 상태를 반환합니다.

### `fsync`

저장소가 `DurabilityMode.FSYNC`을 사용하는 경우 추가 경로는 다음과 같습니다.

- JSONL 항목을 작성합니다.
- 파일 핸들을 플러시합니다.
- `fsync`에 전화
- 허용된 쓰기에 대해 `fsynced` 내구성 상태를 반환합니다.

중요한 점은 내구성이 "아마도 작성되었다"는 막연한 개념이 아니라는 점입니다. 이는 추가 결과의 명시적인 부분입니다.

## 내구성이 수용과 별개인 이유

언뜻 보면 "쓰기 성공"이라는 부울이 필요할 수 있습니다. `Stream`은 수용성과 내구성이 동일한 질문이 아니기 때문에 의도적으로 그 이상을 제공합니다.

발신자는 다음 사항에 관심을 가질 수 있습니다.

- 기록이 정식 역사가 되었는지 여부
- 쓰기가 더 강한 `fsync` 경계를 넘었는지 여부
- 배치가 부분적으로 승인되었는지 여부

이는 서로 관련되어 있지만 별개의 운영 사실입니다.

## 간단한 단일 레코드 예

```python
result = store.append(valid_record)

if not result.success:
    raise RuntimeError(result.rejected)

receipt = result.accepted[0]
print(receipt.sequence)
print(receipt.segment_id)
print(receipt.offset)
print(receipt.durability_status)
```

이는 호출자가 배치와 내구성을 모두 인지하도록 하기 때문에 좋은 기본 패턴입니다.

## 간단한 배치 예

```python
result = store.append_many([record_a, record_b, record_c])

print("accepted:", result.accepted_count)
print("rejected:", result.rejected_count)

for rejection in result.rejected:
    print("rejection:", rejection)

for receipt in result.accepted:
    print(receipt.sequence, receipt.record_ref, receipt.durability_status)
```

이 패턴은 잘못된 입력에 대한 가시성을 잃지 않고 유효한 정식 기록을 보존하려는 경우에 유용합니다.

## 쓰기 경로의 일반적인 실수

### 유효성 검사를 선택적인 입력 위생으로 처리

검증은 부드러운 제안이 아닙니다. 값이 정식 기록이 될 수 있는지 여부를 결정하는 문입니다.

### `append_many()`이 트랜잭션이라고 가정

그렇지 않습니다. 다른 항목이 실패하더라도 유효한 항목이 계속 허용될 수 있습니다.

### `durability_status` 무시 중

운영 모델이 더 강력한 지속성 보장에 관심이 있는 경우 모든 것을 "작성"으로 축소하는 대신 반환된 내구성 정보를 읽어야 합니다.

### 게재위치 메타데이터를 부수적으로 처리

`sequence`, `segment_id` 및 `offset`은(는) 화장품이 아닙니다. 쓰기가 정식 기록에 들어간 위치와 나중에 읽기, 재생 및 복구 동작에 대한 전원을 설명합니다.

### 도우미 인덱스 쓰기가 중요한 부분이라고 가정

그렇지 않습니다. 표준 행위는 세그먼트 항목을 추가하는 것입니다. 지수 업데이트는 파생됩니다.

## 성공적인 쓰기의 작동 의미

성공적인 쓰기는 다음을 의미합니다.

- 입력이 올바르게 정규화되었습니다.
- 표준 유효성 검사 규칙이 통과되었습니다.
- 세그먼트 항목이 작성되었습니다.
- 진행 상황 추가 고급
- 게재위치 메타데이터가 할당되었습니다.
- 파생 인덱스 포인터가 추가되었습니다.

다음을 의미하지 **않습니다**:

- 전체 상점에는 이전 손상이 없습니다.
- 호출자는 무결성 검사를 영원히 무시할 수 있습니다.
- 관련 없는 역사적 이유로 인해 이후 재생이 실패할 수 없습니다.

쓰기 경로 페이지는 거짓된 편안함이 아닌 정확한 자신감을 가르쳐야 하기 때문에 이는 중요합니다.

## 거부된 쓰기의 작동 의미

거부된 쓰기는 입력이 정식 기록에 허용되지 않았음을 의미합니다. 대개는 올바른 결과입니다.

실제로 거부는 다음으로부터 상점을 보호합니다.

- 잘못된 정식 봉투
- 잘못된 타임스탬프
- 지원되지 않는 스키마 버전
- 선택한 레코드 유형과 일치하지 않는 페이로드
- 잘못된 상관 관계 또는 안정적인 참조 필드

중요한 점은 거절은 신뢰할 수 있는 매장을 유지하기 위한 일부이지, 도서관이 불필요하게 엄격하다는 증거가 아니라는 점입니다.

## 좋은 쓰기 측 체크리스트

쓰기 경로에서 이동하기 전에 이제 정신 모델에 다음이 모두 포함되어 있는지 확인하는 것이 좋습니다.

1. 허용된 쓰기와 지속성 쓰기는 서로 관련되어 있지만 별개의 개념입니다.
2. `append_many()`은(는) 부분적으로 성공할 수 있습니다.
3. 검증은 표준 기록을 보호하는 것입니다
4. 정렬된 기록이 중요하기 때문에 게재위치 메타데이터가 중요합니다.
5. 도우미 인덱스는 대신에 정식 쓰기 후에 새로 고쳐집니다.
6. 성공적인 추가는 이후 무결성 검사의 필요성을 대체하지 않습니다.
