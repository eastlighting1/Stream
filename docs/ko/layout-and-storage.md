# 레이아웃과 저장소

이 페이지에서는 `Stream`이(가) 디스크에 배치되는 방식, 각 파일 계열의 의미, 레이아웃의 어떤 부분이 표준인지 파생인지, 운영자가 매니페스트 상태, 세그먼트 파일, 도우미 인덱스, 시퀀스 번호, 오프셋, 세그먼트 롤오버, 재구축 동작 및 복구 동작을 해석하는 방법을 설명합니다.

`Mental Model`이 신뢰 경계를 개념적으로 정의하는 경우 이 페이지는 파일 시스템에서 해당 경계가 어디에 있는지 정확하게 보여줍니다.

## 이 페이지가 결정하는 데 도움이 되는 것

이 페이지를 읽고 나면 다음 질문에 명확하게 답할 수 있습니다.

- 디스크의 기본 저장소 레이아웃은 어떻게 생겼나요?
- `manifest.json`의 책임과 책임이 없는 것
- `segments/`에 포함된 내용과 표준인 이유
- `indexes/`에 포함된 내용과 파생된 이유
- 저장 측면에서 `sequence`, `segment_id` 및 `offset`의 의미
- 세그먼트 롤오버 작동 방식
- 재구축 및 복구 작업 중 변경되는 사항
- 로컬 저장소를 디버깅할 때 먼저 신뢰해야 하는 파일

## 레이아웃에 자체 페이지가 필요한 이유

언뜻 보기에 `Stream` 저장소는 간단해 보일 수 있습니다. 하나의 로컬 루트 아래에 몇 개의 JSON 파일이 있습니다. 실제로 레이아웃은 매장의 철학이 드러나는 곳입니다.

- 정식 기록은 검사 가능해야 합니다.
- 추가 진행 상황은 명시적이어야 합니다.
- 도우미 상태는 다시 빌드 가능해야 합니다.
- 로컬 파일에서 손상을 진단할 수 있어야 합니다.

이는 디스크상의 모양이 부수적인 포장이 아니라는 것을 의미합니다. 이는 정상적인 작동과 손상 복구 중에 라이브러리를 이해할 수 있게 유지하는 방법의 일부입니다.

## 한 문장 저장 모델

`Stream`은(는) JSONL 세그먼트 파일에 정식 추가 기록을 저장하고, 매니페스트에서 쓰기 프론티어 상태를 추적하며, 정식 세그먼트에서 다시 빌드할 수 있는 파생 JSONL 인덱스에 실제 스캔 포인터를 유지합니다.

문장이 자연스러워지면 디스크에 있는 모든 것이 더 의미가 있습니다.

## 기본 매장 레이아웃

`.stream-store`에 뿌리를 둔 건강한 지역 상점은 대략 다음과 같습니다.

```text
.stream-store/
  manifest.json
  segments/
    segment-000001.jsonl
    segment-000002.jsonl
  indexes/
    run_ref/
      run__eval-1.jsonl
    stage_execution_ref/
      stage__evaluate.jsonl
    record_type/
      structured_event.jsonl
      metric.jsonl
```

어떤 경우에는 다음과 같은 내용도 볼 수 있습니다.

```text
.stream-store/
  quarantine/
    segment-000003.jsonl
```

`quarantine/` 디렉터리는 복구 작업에서 정식 파일을 다시 쓰기 전에 손상된 세그먼트의 다듬기 전 복사본을 보존해야 할 때 나타납니다.

## 최상위 구조

최상위 수준에서 상점은 의도적으로 몇 가지 명확하게 다른 책임으로 분할됩니다.

### `manifest.json`

매니페스트 트랙은 프론티어와 레이아웃 상태를 추가합니다.

### `segments/`

세그먼트는 정식 추가 기록을 보유합니다.

### `indexes/`

인덱스는 일반 스캔을 가속화하는 데 사용되는 파생 포인터를 보유합니다.

### `quarantine/`

검역소는 특정 복구 작업 전에 손상된 세그먼트의 보존된 복사본을 보관합니다.

해당 분할은 신뢰 분할을 반영하므로 중요합니다.

- 세그먼트는 진실이다
- 매니페스트는 현재 쓰기 진행 상황을 조정합니다.
- 인덱스는 도우미입니다
- 격리는 수리측 안전 인공물입니다.

## 레이아웃을 검사할 수 있는 이유

`Stream`은(는) 일반 로컬 도구로 읽을 수 있는 레이아웃을 의도적으로 사용합니다.

이는 다음을 의미합니다.

- 텍스트 편집기로 세그먼트를 열 수 있습니다
- 사용자 정의 디버거 없이 매니페스트 드리프트를 검사할 수 있습니다.
- 디렉토리 트리를 보면 인덱스가 존재하는지 여부를 알 수 있습니다.
- 구체적인 파일 측면에서 수리 결정을 설명할 수 있습니다.

이는 로컬 관측성 스토리지가 불투명해지면 신뢰하기 가장 어렵기 때문에 중요합니다. `Stream`은(는) 안전 모델의 일부로 검사 가능성을 선택합니다.

## `manifest.json`은(는) 무엇인가요?

`manifest.json`은 상점의 로컬 작동 상태 파일입니다.

정상 매니페스트는 다음과 같습니다.

```json
{
  "layout_mode": "jsonl_segments",
  "layout_version": "1",
  "current_segment_id": 2,
  "next_sequence": 48,
  "last_committed_sequence": 47,
  "current_segment_record_count": 19
}
```

매니페스트는 다음과 같은 질문에 답합니다.

- 이 매장은 어떤 레이아웃 모드를 사용하고 있나요?
- 이 매장은 어떤 레이아웃 버전을 사용하고 있나요?
- 현재 추가를 위해 활성화된 세그먼트
- 다음에는 어떤 순서를 할당해야 할까요?
- 가장 최근에 커밋된 시퀀스
- 현재 세그먼트가 현재 보유하고 있는 레코드 수

중요한 점은 `manifest.json`이 저장소가 예상대로 계속 추가하는 데 도움이 된다는 것입니다. 이는 정식 역사를 대체하지 않습니다.

## `manifest.json`이(가) 아닌 것

매니페스트는 다음과 같습니다.

- 기본 레코드 저장소
- 세그먼트 검사 대체
- 정식 역사의 완전한 표현
- 기본 세그먼트가 정상임을 보장합니다.

로컬 스토리지 디버깅에서 가장 쉬운 실수 중 하나가 매니페스트를 마치 진실인 것처럼 취급하는 것이기 때문에 이는 중요합니다. 그렇지 않습니다. 추가 프론티어 상태에 대한 간략한 설명입니다.

## 매니페스트 필드와 그 의미

### `layout_mode`

이는 사용되는 레이아웃 전략을 알려줍니다. 현재 중요한 예상 값은 `jsonl_segments`입니다.

### `layout_version`

이는 현재 레이아웃 버전을 알려줍니다. 호환성과 향후 레이아웃 발전을 고려할 때 이는 중요합니다.

### `current_segment_id`

이는 저장소에서 활성 추가 대상으로 간주하는 세그먼트를 알려줍니다.

### `next_sequence`

이는 다음에 승인된 레코드가 어떤 시퀀스 번호를 받아야 하는지 알려줍니다.

### `last_committed_sequence`

이는 저장소가 커밋된 것으로 기록한 가장 최근 시퀀스를 알려줍니다.

### `current_segment_record_count`

이는 매니페스트 상태에 따라 현재 세그먼트에 현재 포함된 레코드 수를 알려줍니다.

이러한 값은 함께 생각되는 추가 경로를 알려줍니다. 그들은 실제로 어떤 역사가 존재하는지 정의하지 않습니다. 세그먼트가 그렇게합니다.

## 매니페스트 불일치가 중요한 경우

`manifest.json`의 유용한 점 중 하나는 관찰된 세그먼트 기록과 일치하지 않을 수 있다는 것입니다.

그 불일치는 단지 장부 관리 소음이 아닙니다. 작동 신호입니다.

예를 들어 불일치는 다음을 암시할 수 있습니다.

- 중단된 쓰기 경로
- 손상된 매니페스트
- 재계산이 필요한 정식 기록을 수정하거나 다듬었습니다.
- 추가 프론티어 추적과 관찰된 세그먼트 콘텐츠 사이의 드리프트

이는 매니페스트 불일치가 매니페스트 및 세그먼트 기록을 독립적으로 해석하기보다는 함께 해석해야 하는 이유를 보여주는 가장 명확한 예 중 하나이기 때문에 중요합니다.

## `segments/`에 사는 것

`segments/` 디렉터리는 표준 추가 저장소입니다.

각 세그먼트 파일은 줄 기반 JSONL이며, 비어 있지 않은 각 줄은 허용되는 하나의 추가 항목을 나타냅니다. 세그먼트 항목에는 다음이 포함됩니다.

- 정식 `sequence`
- 상점에서 생성된 `appended_at` 타임스탬프
- 표준 `record` 페이로드

개념적 예는 다음과 같습니다.

```json
{
  "sequence": 47,
  "appended_at": "2026-04-03T00:10:15Z",
  "record": {
    "record_ref": "record/eval-metric-2",
    "record_type": "metric",
    "recorded_at": "2026-04-03T00:10:15Z",
    "observed_at": "2026-04-03T00:10:15Z",
    "producer_ref": "scribe.python.local",
    "run_ref": "run/eval-1",
    "stage_execution_ref": "stage/evaluate",
    "operation_context_ref": "op/evaluate-metric-2",
    "correlation_refs": {
      "trace_id": "trace/eval-1"
    },
    "completeness_marker": "complete",
    "degradation_marker": "none",
    "schema_version": "1.0.0",
    "payload": {
      "metric_key": "evaluation.loss",
      "value": 0.39,
      "value_type": "scalar",
      "aggregation_scope": "step",
      "tags": {
        "split": "validation"
      }
    }
  }
}
```

중요한 점은 이 선이 지역적 수용의 표준적인 사실이라는 것입니다. 상점의 다른 모든 것은 그것의 하류에 있습니다.

## 세그먼트가 표준인 이유

세그먼트는 다음을 유지하므로 표준입니다.

- 허용된 레코드 페이로드
- 허용된 추가 순서
- 나중에 읽고 재생하여 해석할 수 있는 내구성 있는 지역 역사

인덱스가 한 가지를 말하고 세그먼트 라인이 다른 것을 말하면 세그먼트 라인이 승리합니다.

매니페스트에는 다음 시퀀스가 ​​`48`이어야 한다고 나와 있지만 세그먼트 기록에는 그렇지 않은 것으로 표시되는 경우 세그먼트 기록이 여전히 진실에 더 가까운 표현입니다.

이러한 비대칭성은 의도적인 것입니다.

## 세그먼트 이름 지정 및 순서

세그먼트는 다음과 같은 이름을 사용합니다.

- `segment-000001.jsonl`
- `segment-000002.jsonl`

패딩된 숫자 접미사는 다음과 같은 이유로 유용합니다.

- 자연스럽게 정렬되네요
- 추가 진행 상황을 시각적으로 쉽게 검사할 수 있습니다.
- 코드와 사람 모두에게 레이아웃 해석을 간단하게 유지합니다.

읽기 및 쓰기 메타데이터의 `segment_id`은 이 숫자 ID에 해당합니다.

## 세그먼트 파일이 롤오버되는 이유

`Stream`은(는) 단일 파일에 영원히 계속해서 쓰지 않습니다. 세그먼트 롤오버는 활성 세그먼트가 구성된 `max_segment_bytes` 임계값에 도달할 때 발생합니다.

실제로 이는 다음을 의미합니다.

- 현재 세그먼트는 크기 제한 미만인 동안 추가 대상으로 유지됩니다.
- 크기 임계값에 도달하면 다음 추가가 새 세그먼트 ID로 이동합니다.
- 추가 시퀀스 번호는 새 파일 경계를 넘어 계속 증가합니다.

여러 세그먼트가 문제가 아닌 일반적이고 건강한 성장의 신호이기 때문에 이는 중요합니다.

## `sequence`의 의미

`sequence`은 허용된 레코드의 전역 추가 순서 ID입니다.

그렇지 않습니다:

- 하나의 세그먼트에만 로컬
- 의미론적 의미가 없는 편의 카운터
- 라인 주문만으로 교체 가능

그것은:

- 매장 전체에 걸쳐 단조로운 분위기
- 리플레이 해석의 중심
- 무결성 검사의 핵심

시퀀스 연속성은 일관된 표준 역사의 가장 명확한 표현 중 하나이기 때문에 이것이 중요합니다.

## `offset`의 의미

`offset`은 세그먼트 파일 내 레코드의 행 위치입니다.

실제로 매장에 다음과 같이 알려줍니다.

- 세그먼트 내부의 특정 라인은 도우미 인덱스 포인터에 해당합니다.
- 저장된 레코드가 세그먼트 컨텍스트 내에서 존재하는 곳

도우미 인덱스는 레코드 페이로드를 복사하지 않기 때문에 이는 중요합니다. `(segment_id, offset)`을 사용하여 표준 세그먼트 위치를 다시 가리킵니다.

## `sequence`과 `offset`이 모두 존재하는 이유

언뜻 보면 `sequence`과 `offset`을 모두 갖는 것이 중복되어 보일 수 있습니다.

그들은 다양한 목적을 가지고 있습니다:

- `sequence`은 저장소 전체에 걸쳐 전역 추가 순서를 표현합니다.
- `offset`은 특정 세그먼트 내의 로컬 배치를 표현합니다.

다음과 같은 이유로 둘 다 필요합니다.

- 재생과 무결성은 글로벌 질서를 중시합니다.
- 색인화된 조회에는 정확한 로컬 파일 위치가 필요합니다.

해당 분할을 확인하면 스토리지 레이아웃을 추론하기가 훨씬 쉬워집니다.

## `indexes/`에 사는 것

`indexes/` 디렉터리에는 파생 JSONL 도우미 파일이 포함되어 있습니다.

가족별로 그룹화되어 있습니다.

- `indexes/run_ref/`
- `indexes/stage_execution_ref/`
- `indexes/record_type/`

각 파일에는 다음과 같은 포인터 항목이 포함되어 있습니다.

```json
{"offset":1,"segment_id":1,"sequence":1}
```

해당 포인터는 실제로 다음과 같이 말합니다.

- 이 일치 레코드는 세그먼트 `1`에 있습니다.
- 온라인 `1`
- 전역 시퀀스 `1` 사용

인덱스는 스캔 작업 범위를 좁히기 때문에 유용합니다. 이는 표준 세그먼트 항목을 대체하므로 유용하지 않습니다.

## 인덱스 파일 이름이 삭제된 것처럼 보이는 이유

`run/eval-1`과 같은 값은 다음과 같은 파일 이름이 됩니다.

- `run__eval-1.jsonl`

이러한 정리 작업은 실용적인 키가 파일 시스템에 안전한 파일 이름이 될 수 있도록 하기 위해 존재합니다. 이는 표준 레코드의 의미 변환이 아닌 저장 편의 세부 사항입니다.

중요한 점은 표준 값이 세그먼트 페이로드에 여전히 존재한다는 것입니다. 파일 이름은 단지 도우미 표현일 뿐입니다.

## 지수가 파생상품인 이유

인덱스는 정식 기록에서 다시 작성될 수 있기 때문에 파생됩니다.

이는 다음을 의미합니다.

- 인덱스 파일을 삭제해도 정식 기록은 삭제되지 않습니다.
- 인덱스 재구축은 세그먼트에서 도우미 포인터를 재구성합니다.
- 손상된 인덱스는 기본적으로 손상된 진실로 해석되어서는 안 됩니다.

이는 운영자가 인덱스 손상과 세그먼트 손상을 매우 다르게 처리해야 하기 때문에 중요합니다.

## 인덱스 재구축 중 발생하는 일

인덱스가 재구축되면 `Stream`:

1. 표준 기록을 스캔합니다.
2. 저장된 세그먼트 항목에서 포인터 레코드를 파생합니다.
3. `indexes/` 아래에 인덱스 파일을 다시 생성합니다.

중요한 점은 재구축은 진실이 아닌 편의성을 회복한다는 점이다.

표준 세그먼트가 이미 손상된 경우 재구축을 통해 읽을 수 없거나 누락된 부분을 마법처럼 다시 만들 수 없습니다. 해석 가능한 상태로 유지되는 헬퍼 포인터만 다시 작성할 수 있습니다.

## 수리 중 일어나는 일

복구는 표준 세그먼트 테일에서 작동할 수 있다는 점에서 재구축과 다릅니다.

알려진 잘린 꼬리 복구 경로 `Stream`에서:

1. 손상된 후행 세그먼트 라인을 감지합니다.
2. 원본 세그먼트를 `quarantine/`에 복사합니다.
3. 표준 세그먼트에서 손상된 꼬리를 다듬습니다.
4. 매니페스트 상태를 다시 계산합니다.
5. 파생 인덱스를 다시 작성합니다.

수리는 정식 스토리지를 변경하는 반면, 재구축은 파생 도우미 상태만 변경하기 때문에 이는 중요합니다.

## `quarantine/`의 의미

`quarantine/` 디렉토리는 수리를 위한 안전 인공물입니다.

손상된 세그먼트가 잘려질 경우 `Stream`은 먼저 `quarantine/` 아래에 복사본을 보존합니다.

이는 다음을 의미합니다.

- 수리는 조용한 파괴가 아니다
- 운영자는 나중에 원래 손상된 바이트를 검사할 수 있습니다.
- 시스템은 수리 중에 변경된 사항에 대한 명시적인 감사 추적을 유지합니다.

이는 숨겨진 정리보다 명시적이고 검사 가능한 작업을 선호하는 라이브러리의 가장 명확한 예 중 하나입니다.

## 매장을 수동으로 검사하는 방법

로컬 저장소를 디버깅할 때 좋은 검사 순서는 다음과 같습니다.

1. `manifest.json` 열어
2. `segments/` 나열 및 검사
3. 영향을 받은 시퀀스 주변의 특정 세그먼트 라인을 검사합니다.
4. 세그먼트 현실을 이해한 후에만 `indexes/`을 검사하십시오.
5. 이미 수리가 이루어졌는지 확인하려면 `quarantine/`을(를) 확인하세요.

이 순서는 신뢰 경계를 유지하기 때문에 중요합니다.

- 정식 역사 먼저
- 도우미 상태 두 번째
- 마지막 복구 아티팩트

## 레이아웃 해석의 일반적인 실수

### 매니페스트를 전체 저장소로 취급

매니페스트는 유용하지만 표준 레코드 기록은 아닙니다.

### 읽기 쉽기 때문에 인덱스를 신뢰할 수 있는 것으로 취급

인덱스는 간결하고 읽기 쉽지만 여전히 파생 도우미 상태입니다.

### 여러 세그먼트를 문제로 취급

다중 세그먼트 파일은 일반적으로 추가 및 롤오버 규칙에 따라 건전하게 증가합니다.

### 재구성 및 복구가 동일한 종류의 파일을 터치한다고 가정

그렇지 않습니다. 재구축은 파생 인덱스에서 작동합니다. 복구는 정식 세그먼트를 변경한 다음 나중에 도우미 상태를 다시 빌드할 수 있습니다.

### `quarantine/` 무시 중

복구가 실행된 경우 격리 복사본은 운영 스토리의 일부입니다. 어떤 표준 파일이 변경되었는지 알려주고 사전 트림 증거를 보존합니다.

## 건강한 레이아웃의 작동적 의미

건강한 레이아웃은 일반적으로 다음을 의미합니다.

- 세그먼트를 읽을 수 있습니다
- 순서가 일관적이다
- 매니페스트 프론티어가 관찰된 세그먼트 기록과 일치합니다.
- 활성화되면 인덱스가 존재하고 실제 세그먼트 라인을 다시 가리킵니다.

이는 모든 파일이 동일한 방식으로 완벽해야 한다는 의미는 아닙니다. 이는 스토리지 계층이 수행해야 하는 역할에 동의한다는 의미입니다.

## 손상된 레이아웃의 작동 의미

손상된 레이아웃은 다음과 같이 나타날 수 있습니다.

- 잘못된 세그먼트 선
- 잘린 꼬리
- 서열 공백
- 명백한 표류
- 누락되거나 오래된 파생 인덱스

중요한 점은 모든 손상의 심각도가 동일하지는 않다는 것입니다.

- 손상된 인덱스는 불편할 수 있습니다
- 손상된 세그먼트가 심각합니다
- 손상된 매니페스트 상태는 중요하지만 여전히 표준 세그먼트 현실에 부차적입니다.

이러한 구별이 바로 `Stream`이(가) 이러한 방식으로 이러한 파일을 분할하는 이유입니다.

## 좋은 스토리지 측 체크리스트

계속 진행하기 전에 이제 스토리지 정신 모델에 다음이 모두 포함되어 있는지 확인하는 것이 좋습니다.

1. `segments/`은 표준입니다.
2. `manifest.json`은 기록을 교체하는 대신 추가 프론티어를 추적합니다.
3. `indexes/`은 파생 포인터 파일입니다.
4. `sequence`은 전역이고 `offset`은 세그먼트에 대해 로컬입니다.
5. 세그먼트 롤오버는 정상입니다.
6. 재구축은 도우미 상태를 복원합니다.
7. 수리는 표준 꼬리를 변경하고 `quarantine/`의 원본을 보존할 수 있습니다.
