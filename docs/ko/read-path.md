# 읽기 경로

이 페이지에서는 `Stream`이 저장된 기록을 다시 읽는 방법, `scan`과 `replay`이 동일하지 않은 이유, 도우미 인덱스가 읽기 속도를 높일 수 있는 경우, `strict`, `tolerant`, 경고, 알려진 간격 및 저장소의 신뢰 경계를 넘지 않고 내보내기 동작을 해석하는 방법에 대해 설명합니다.

쓰기 경로가 정식 기록이 들어오는 방식을 설명한다면, 읽기 경로는 해당 기록이 디스크에 저장된 후 해석되어야 하는 방식을 설명합니다.

## 이 페이지가 결정하는 데 도움이 되는 것

이 페이지를 읽고 나면 다음 질문에 명확하게 답할 수 있습니다.

- `scan()` 사용 시기 및 `replay()` 사용 시기
- 스캔이 헬퍼 인덱스를 사용할 수 있는 경우와 사용할 수 없는 경우
- `strict` 및 `tolerant` 재생이 실제로 의미하는 것
- `warnings`과 `known_gaps`이(가) 당신에게 말하는 것
- `export_jsonl()`이 진실을 재정의하는 대신 재생과 어떻게 관련되는지
- 성공적인 읽기가 항상 안전한 해석과 같지는 않은 이유

## 읽기 경로에 자체 페이지가 필요한 이유

언뜻 보기에 `Stream`에서 읽는 것은 간단해 보일 수 있습니다. 기록을 요청하면 상점에서 기록을 반환합니다. 실제로 읽기 경로는 `Stream`이 저장된 표준 기록을 운영자 결정으로 바꾸는 곳입니다.

이것이 바로 읽기 경로에 둘 이상의 표면이 필요한 이유입니다.

- `scan()` 직접 검사용
- 무결성 인식 기록 해석을 위한 `replay()`
- 재생 의미론을 기반으로 구축된 다운스트림 JSONL 출력의 경우 `export_jsonl()`

중요한 점은 독서가 단순한 검색이 아니라는 점이다. 해석이기도 합니다.

## 한 문장 읽기 모델

`Stream`은(는) 저장된 레코드를 직접 스캔하거나 명시적인 무결성 의미론에 따라 재생하여 표준 기록을 읽습니다. 선택적으로 파생 색인을 사용하여 표준 소스를 변경하지 않고 실제 스캔 범위를 좁힙니다.

이 문장에는 전체 읽기 측 철학이 포함되어 있습니다.

- 정식 역사는 여전히 세그먼트에 존재합니다.
- 도우미 색인은 관련 줄을 찾는 데 도움이 될 수 있습니다.
- 재생은 검색 위에 안전 의미를 추가합니다.

## 세 가지 주요 읽기 표면

`Stream`은 밀접하게 관련되어 있지만 별개의 세 가지 읽기 측 표면을 노출합니다.

1. `scan()`
2. `replay()`
3. `export_jsonl()`

이것들은 서로 연관되어 있지만 같은 의미는 아닙니다.

### `scan()`

기록을 직접 저장하려면 `scan()`을(를) 사용하세요.

### `replay()`

저장소 무결성 및 재생 모드를 명시적으로 설명하는 읽기를 원할 경우 `replay()`을(를) 사용하세요.

### `export_jsonl()`

다른 도구에 줄 기반 JSON 출력의 재생 기록이 필요한 경우 `export_jsonl()`을 사용하세요.

중요한 점은 내보내기가 우회가 아닌 재생을 기반으로 구축된다는 것입니다.

## `scan()`이(가) 하는 일

`scan()`은(는) 직접 저장된 기록 검사 경로입니다.

실제로는 다음과 같습니다.

- 추가 순서로 세그먼트 기록을 살펴봅니다.
- 요청된 `ScanFilter`을 적용합니다.
- `StoredRecord` 값을 반환합니다.
- 몇몇 간단한 경우에는 도우미 인덱스를 사용할 수 있습니다.

간단한 예는 다음과 같습니다.

```python
from stream import ScanFilter

records = list(store.scan(ScanFilter(run_ref="run/eval-1")))

for stored in records:
    print(
        stored.sequence,
        stored.segment_id,
        stored.offset,
        stored.record["record_type"],
    )
```

`scan()`은(는) 최선의 방법으로 의도적으로 지루하기 때문에 이것이 중요합니다. "이 필터와 일치하는 저장된 레코드는 무엇입니까?"라고 묻고 싶을 때 사용하는 읽기 경로입니다.

## `StoredRecord`이 나타내는 것

`StoredRecord`은 원래 레코드 매핑이 아닙니다. 여기에는 다음이 모두 포함됩니다.

- 표준 레코드 데이터
- 추가측 배치 메타데이터

즉, 각 `StoredRecord`에는 다음이 포함됩니다.

- `sequence`
- `segment_id`
- `offset`
- `appended_at`
- `record`

`Stream`의 읽기는 다음 간의 구별을 유지하므로 이는 중요합니다.

- 기록이 말하는 것
- 지역 정식 역사에 들어간 장소와 시기

이러한 구별은 재생 해석과 이후의 운영 디버깅에 매우 중요합니다.

## `ScanFilter`이(가) 실제로 무엇을 하고 있나요?

`ScanFilter`은 일반적인 검사 패턴에 대한 실용적인 읽기 측 필터 개체입니다.

다음을 기준으로 필터링할 수 있습니다.

- `run_ref`
- `stage_execution_ref`
- `record_type`
- `start_time`
- `end_time`

언뜻 보면 이것은 작은 편의 포장지처럼 보일 수 있습니다. 실제로는 라이브러리의 독자적인 읽기 형태를 포착합니다. `Stream`은 임의의 관계형 쿼리가 아닌 실제 로컬 기록 검사에 최적화되어 있습니다.

## `scan()`이 하지 않는 일

`scan()`이 약속하지 않는 것이 무엇인지 이해하는 것도 마찬가지로 중요합니다.

`scan()`은(는) 다음을 수행하지 않습니다.

- 매장이 정상인지 자동으로 인증
- 재생 경고 및 알려진 공백을 첨부하세요.
- 재생과 같은 방식으로 손상을 부분 기록으로 재해석합니다.
- 도우미 인덱스를 정보 소스 상태로 승격

깔끔한 `scan()` 결과가 매장이 전 세계적으로 건강하다는 증거로 해석되어서는 안 되기 때문에 이는 중요합니다.

## `scan()`이 도우미 색인을 사용할 수 있는 경우

가장 유용한 읽기 경로 세부 정보 중 하나는 필터가 충분히 단순할 때 `scan()`이 도우미 인덱스를 참조할 수 있다는 것입니다.

실제로 다음과 같은 경우 인덱스 지원 스캔을 사용할 수 있습니다.

- 기간 필터가 없습니다.
- 정확히 하나의 인덱싱된 패밀리가 채워집니다.

이는 이와 같은 필터가 인덱스를 사용할 수 있음을 의미합니다.

```python
ScanFilter(record_type="metric")
```

이와 같은 필터는 인덱스를 사용할 수 있습니다.

```python
ScanFilter(run_ref="run/eval-1")
```

하지만 이와 같은 필터는 다음을 수행할 수 없습니다.

```python
ScanFilter(run_ref="run/eval-1", record_type="metric")
```

시간 경계가 있는 필터는 다음 중 하나를 수행할 수 없습니다.

```python
ScanFilter(run_ref="run/eval-1", start_time="2026-04-03T00:00:00Z")
```

중요한 점은 헬퍼 인덱스가 진실 소스를 재정의하는 것이 아니라 실질적인 범위를 좁히는 데 사용된다는 것입니다.

## 인덱스 스캔이 신뢰 경계를 깨지 않는 이유

언뜻 보기에 읽기 경로에서 인덱스를 사용하는 것은 인덱스를 신뢰할 수 있는 것으로 취급하는 것처럼 보일 수 있습니다. 그런 일은 일어나지 않습니다.

지수는 다음과 같은 역할을 합니다.

- 후보 `(segment_id, offset)` 포인터

세그먼트는 여전히 다음과 같은 기여를 하고 있습니다.

- 실제 표준 줄 내용

따라서 파생 인덱스가 조회를 보다 효율적으로 만드는 경우에도 읽기 경로는 표준 세그먼트에 기반을 두고 있습니다.

이러한 구별은 디자인의 가장 건강한 부분 중 하나입니다.

## `replay()`이(가) 하는 일

`replay()`은 명시적 재생 모드에서 저장된 기록을 읽습니다.

간단한 예는 다음과 같습니다.

```python
from stream import ReplayMode, ScanFilter

replay = store.replay(
    ScanFilter(run_ref="run/eval-1"),
    mode=ReplayMode.STRICT,
)

print(replay.record_count)
print(replay.mode)
print(replay.warnings)
print(replay.known_gaps)
```

`scan()`과 달리 `replay()`은 단순히 일치하는 저장된 레코드를 반환하는 것이 아닙니다. 또한 진실성이 해당 역사를 해석하는 능력에 어떤 영향을 미쳐야 하는지에 대한 진술을 하고 있습니다.

## `scan()`이 이미 존재하는데 `replay()`이 존재하는 이유

이는 읽기 경로에서 가장 큰 개념적 요점 중 하나입니다.

언뜻 보면 `scan()` 및 `replay()`은 둘 다 레코드를 반환하므로 중복되어 보일 수 있습니다. 실제로 그들은 다양한 질문에 대답합니다.

- `scan()`이 "이 필터와 일치하는 저장된 레코드는 무엇입니까?"라고 묻습니다.
- `replay()`이(가) "이 재생 모드에서는 어떤 기록을 안전하게 해석할 수 있나요?"라고 묻습니다.

이는 재생이 안전 의미론이 명시되는 곳임을 의미합니다.

이러한 구별을 염두에두면 나중에 손상된 매장 동작이 놀랍지 않고 일관되게 느껴집니다.

## `strict` 재생

`strict` 재생은 보수적인 읽기 경로입니다.

실제로 `strict` 재생은 다음을 의미합니다.

- 먼저 매장 무결성을 확인하세요
- 저장소가 손상된 경우 진행하는 대신 중지하세요.
- 손상된 역사를 마치 전체인 것처럼 해석하지 마십시오.

`strict`은 "더 까다로운 편의 설정"이 아니기 때문에 이는 중요합니다. 안전 경계입니다.

`strict` 재생이 실패하면 해당 실패가 올바른 운영 결과인 경우가 많습니다.

## `tolerant` 재생

`tolerant` 재생은 손상 인식 읽기 경로입니다.

실제로 `tolerant` 재생은 다음을 의미합니다.

- 먼저 매장 무결성을 확인하세요
- 아직 해석될 수 있는 내용을 계속 읽으세요.
- 필요한 경우 읽을 수 없는 줄을 건너뜁니다.
- 호출자가 기록이 불완전하다는 것을 알 수 있도록 경고 및 알려진 공백을 반환합니다.

`tolerant`이(가) 매장이 건강한 척하는 것이 아니기 때문에 이는 중요합니다. 사용 가능한 부분 기록을 추출하는 동시에 명시적으로 피해를 인정합니다.

## `ReplayResult`이(가) 당신에게 말하는 것

`ReplayResult`은 저장된 레코드의 일반 목록보다 더 풍부한 개체입니다.

여기에는 다음이 포함됩니다.

- `records`
- `record_count`
- `mode`
- `warnings`
- `known_gaps`

중요한 점은 `ReplayResult`이 검색과 무결성 해석을 결합한다는 것입니다.

### `records`

재생된 `StoredRecord` 값입니다.

### `record_count`

선택한 모드에서 재생된 레코드 수를 알려줍니다.

### `mode`

이는 읽기가 `strict` 또는 `tolerant` 의미 체계에 따라 수행되었는지 여부를 알려줍니다.

### `warnings`

허용 재생에서 `warnings`은(는) 반환된 기록을 해석하는 방법에 영향을 미치는 무결성 문제를 요약합니다.

### `known_gaps`

허용 재생에서 `known_gaps`은 읽기 가능한 기록에 구멍을 남긴 무결성 오류에 대해 기계가 읽을 수 있는 설명을 제공합니다.

## `warnings`의 의미

경고는 일반적인 권고 문자열이 아닙니다. 반환된 기록을 주의해서 해석해야 한다는 뜻입니다.

실제로 경고에는 다음이 반영될 수 있습니다.

- 잘못된 JSON 라인
- 잘린 꼬리
- 서열 공백
- 명백한 표류

중요한 점은 경고가 단순한 로깅이 아닌 해석에 속한다는 것입니다.

발신자가 재생 경고를 무시하면 명시적인 안전보다 편리함을 선택하는 것입니다.

## `known_gaps`의 의미

알려진 간격은 관용적 재생의 가장 중요한 결과 중 하나입니다.

`tolerant` 모드는 일부 기록이 안전하지 않거나 읽을 수 없게 되었다는 사실을 숨기려고 하지 않기 때문에 존재합니다. 대신 다음과 같이 말합니다.

- 이 기록은 재생되었습니다
- 이러한 손상된 위치는 그렇지 않았습니다.

이는 관용적인 재생 결과가 불완전성을 허용하기 때문에 유용하다는 것을 의미합니다.

다운스트림 도구는 부분적인 기록을 완전한 진실과 혼동하지 않는 한 여전히 유용하게 검사할 수 있기 때문에 이는 운영상 중요합니다.

## Replay가 무결성을 먼저 확인하는 이유

재생 경로는 계속 방법을 결정하기 전에 의도적으로 무결성 검사를 실행합니다.

재생은 단순히 "가능한 한 모든 줄을 읽는" 것이 아니기 때문에 이는 중요합니다. "가게의 무결성 상태에 맞게 이력을 읽는 것"입니다.

실제로:

- `strict` 재생이 손상된 저장소에서 진행을 거부합니다.
- `tolerant` 재생은 계속되지만 명시적인 경고와 알려진 공백만 있습니다.

이것은 `Stream`이 조용한 편리함보다 명시적인 안전함을 선택한 가장 분명한 예 중 하나입니다.

## 부패는 어떻게 되나요?

손상된 저장소 읽기 경로를 명시적으로 생각하는 것이 도움이 됩니다.

### `scan()`에 전화하는 경우

`scan()`은(는) 여전히 일치하는 레코드를 반환할 수 있지만 기본 안전 표면은 아닙니다.

읽고 있는 경로에 손상이 나타나는 경우:

- 일반 스캔에서는 읽을 수 없는 줄이 발생할 수 있습니다.
- 해당 컨텍스트에서 손상 허용이 활성화되지 않은 경우 읽기 속도가 높아질 수 있습니다.
- 결과에는 재생 의미 체계가 포함되지 않습니다.

### `replay(mode="strict")`에 전화하는 경우

저장소가 손상된 것으로 분류되면 재생이 일찍 실패합니다.

### `replay(mode="tolerant")`에 전화하는 경우

재생이 계속 시도되고 다음이 반환됩니다.

- 아직 해석 가능한 기록
- 무결성 문제를 설명하는 경고
- 손상이 관찰된 위치를 설명하는 알려진 간격

이는 읽기 경로가 정상으로 보이는 최선의 읽기 뒤에 숨기려고 하는 대신 손상에 대해 의도적으로 명시적이기 때문에 중요합니다.

## `iter_replay()`의 용도

`iter_replay()`은 `replay()`과 동일한 개념적 읽기 경로에 대해 존재하지만 최종 튜플을 먼저 빌드하는 대신 반복자로 레코드를 생성합니다.

다음과 같은 경우에 사용하세요:

- 당신은 재생 의미를 원한다
- 당신은 점진적인 소비를 원합니다
- 완전히 구체화된 재생 결과가 즉시 필요하지 않습니다.

멘탈 모델은 여전히 ​​동일합니다. 의미론을 먼저 재생하고 반복 스타일을 두 번째로 사용합니다.

## `export_jsonl()`이(가) 하는 일

`export_jsonl()`은 재생을 기반으로 하는 읽기 측 편의 기능입니다.

간단한 예는 다음과 같습니다.

```python
from pathlib import Path

from stream import ReplayMode, ScanFilter

result = store.export_jsonl(
    Path("exports/eval.jsonl"),
    ScanFilter(run_ref="run/eval-1"),
    mode=ReplayMode.STRICT,
)

print(result.record_count)
```

중요한 점은 내보내기가 새로운 해석 레이어를 생성하지 않는다는 것입니다. 그것:

1. 선택한 모드에서 재생을 수행합니다.
2. 재생된 레코드 페이로드를 JSONL로 작성합니다.
3. 호출자가 해석 메타데이터를 계속 검사할 수 있도록 동일한 재생 결과를 반환합니다.

내보낸 JSONL은 새로운 정식 진실 소스가 아니라 편리한 출력이기 때문에 이는 중요합니다.

## 내보내기가 읽기 측에 존재하는 이유

수출 자체를 저장 문제로 생각하고 싶을 수도 있습니다. `Stream`에서는 읽기 경로 확장으로 보는 것이 더 좋습니다.

그 이유는 내보내기 때문입니다.

- 재생 의미론에 따라 다름
- 권위 있는 새 역사를 만들지 않는다
- 이미 해석된 기록을 다른 도구에 전달하기 위해 존재합니다.

이런 방식으로 내보내기를 보면 내보내기 전에 재생 경고가 여전히 중요한 이유가 분명해집니다.

## 건전한 읽기 예

정상적인 읽기 세션은 일반적으로 다음과 같습니다.

```python
records = list(store.scan(ScanFilter(record_type="metric")))
replay = store.replay(ScanFilter(record_type="metric"), mode=ReplayMode.STRICT)

print(len(records))
print(replay.record_count)
print(replay.warnings)
```

건강한 매장에서는:

- 스캔 횟수와 재생 횟수가 일치할 수 있습니다.
- 경고가 비어 있습니다
- 알려진 공백이 비어 있습니다.
- 엄격한 재생 성공

이는 `scan()`과 `replay()`을 서로 바꿔서 사용할 수 있다는 의미는 아닙니다. 이는 그들이 건강한 조건에서 동의한다는 것을 의미합니다.

## 관대 한 읽기 예

허용되는 읽기 세션은 개념적으로 다음과 같습니다.

```python
replay = store.replay(mode=ReplayMode.TOLERANT)

print(replay.record_count)
print(replay.warnings)
print(replay.known_gaps)
```

손상된 매장에서는 다음이 발생할 수 있습니다.

```text
record_count == 17
warnings == ("last segment line looks truncated or partially written", ...)
known_gaps == ("truncated_line@segment=3:line=42", ...)
```

중요한 점은 결과가 여전히 유용하지만 해석에 경고를 첨부하는 경우에만 가능하다는 것입니다.

## 읽기 경로의 일반적인 실수

### `scan()` 및 `replay()`을 상호 교환 가능한 것으로 취급

정상적인 경우에는 동일한 기록을 반환할 수 있지만 다른 결정을 위해 존재합니다.

### 재생 경고 무시

경고는 소음이 아닙니다. 반환된 기록을 완전히 건강한 것으로 읽어서는 안 된다는 진술입니다.

### `tolerant`을 더 나은 기본값으로 처리

`tolerant`은(는) 더 유능한 `strict`이 아닙니다. 명시적인 불완전성을 허용하는 것은 다른 절충안입니다.

### 인덱스 스캔을 가정하면 인덱스가 정식임을 의미합니다.

인덱스 스캔은 인덱스를 사용하여 검색 범위를 좁힌 다음 여전히 세그먼트 라인을 진실로 읽습니다.

### 내보낸 JSONL을 새로운 사실로 취급

내보낸 파일은 다운스트림 편의 아티팩트입니다. 이는 표준 세그먼트 기록을 대체하지 않습니다.

## 성공적인 읽기의 작동적 의미

성공적인 읽기는 표면에 따라 다른 의미를 가질 수 있습니다.

### `scan()` 성공

이는 일치하는 레코드를 성공적으로 읽었음을 의미합니다. 글로벌 무결성을 인증하지 않습니다.

### `strict` 재생 성공

이는 저장소의 무결성 상태가 해당 재생에 대해 엄격한 해석을 허용했음을 의미합니다.

### `tolerant` 재생 성공

이는 사용 가능한 부분 기록이 반환되었음을 의미하며, 저장소가 정상이라는 의미는 아닙니다.

이러한 구별이 중요한 이유는 동일한 단어인 "성공"이 이러한 표면을 함께 축소하면 매우 다른 작동 의미를 숨길 수 있기 때문입니다.

## 좋은 읽기 측 체크리스트

계속 진행하기 전에 이제 읽기 측 멘탈 모델에 다음이 모두 포함되어 있는지 확인하는 것이 좋습니다.

1. `scan()`은 직접 검사 경로입니다.
2. `replay()`은 무결성 인식 해석 경로입니다.
3. 인덱스 스캔은 진실 소유권을 변경하지 않고도 편의성을 향상시킵니다.
4. `strict` 부패로 인한 실패는 종종 올바른 결과입니다
5. `tolerant` 재생은 경고와 알려진 공백을 명시적으로 유지하므로 유용합니다.
6. 내보내기는 우회하는 것이 아니라 재생을 기반으로 구축됩니다.
